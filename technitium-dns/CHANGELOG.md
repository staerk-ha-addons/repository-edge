- 🐛 Fix Ingress for host-network add-on via nginx proxy (#42)

## Description

The Ingress panel showed **"refused to connect"** (reported by @pbathuk
on #31, reproduced on a live instance). Root cause: the add-on uses
`host_network: true` with a **fixed** `ingress_port: 5380`. Per HA's
docs, host-network add-ons must use a **dynamic** ingress port
(`ingress_port: 0`) — the Supervisor's ingress proxy can't reach a fixed
port on a host-network container, so it gets connection-refused. The
add-on itself was always healthy; only the ingress path was
misconfigured.

## Fix

- `config.yaml`: `ingress_port: 5380` → **`0`** (dynamic).
- New **nginx** reverse-proxy service that reads the Supervisor-assigned
port at runtime (`bashio::addon.ingress_port`) and proxies it →
Technitium on `127.0.0.1:5380`.
- Technitium stays on `5380`, so **direct access at `http://<host>:5380`
keeps working** (what users rely on today).

## Files
- `Dockerfile`: install `nginx-full` (includes the `map` module for
WebSocket upgrades)
- `rootfs/etc/nginx/nginx.conf`: minimal proxy config + WebSocket `map`
- `rootfs/.../s6-rc.d/ingress/*`: new longrun service that renders the
server block with the dynamic port and runs nginx
- `config.yaml`: dynamic ingress port

## Verification (local)
- ✅ Image builds with `nginx-full` (nginx 1.26.3)
- ✅ `nginx -t` passes with the rendered server block (confirms
`map`/`$connection_upgrade` available)
- ✅ Proxy serves the console: `http://127.0.0.1:<proxy>` → HTTP 200,
`<title>Technitium DNS Server</title>`
- ✅ Direct `:5380` still returns HTTP 200
- ⏳ **Needs on-device check:** the HA dynamic-port read can only be
exercised on a real Supervisor — please confirm the Ingress panel loads
after installing this build.

- 👷 Relax yamllint comment spacing for Renovate action pins (#43)

## Description

After enabling `helpers:pinGitHubActionDigests` (#40), Renovate opened
#41 to SHA-pin the GitHub Actions. That PR **fails YAMLLint**: Renovate
writes pins as `@<sha> # <version>` with a **single** space before the
comment, but `.yamllint` required `min-spaces-from-content: 2`, so every
pinned line trips `too few spaces before comment`.

## Fix

Set `comments.min-spaces-from-content` to **1**. This:
- Accepts Renovate's single-space pin comments (unblocks #41 and all
future digest bumps).
- Matches **Prettier**, which normalizes inline comments to one leading
space — so the two linters stop conflicting (previously we had to work
around this with standalone comments).
- Doesn't regress: existing 2-space comments still satisfy a *minimum*
of 1.

- 👷 Make Renovate reliably auto-upgrade the repo (#40)

## Description

Renovate already had auto-merge fully enabled (`automerge`,
`platformAutomerge`, squash, no dashboard approval), but two things
blocked it in practice. This removes both.

### 1. Remove `.github/dependabot.yaml`
Dependabot was configured for `github-actions`, `docker`, and
`devcontainers` — all of which Renovate already handles. Dependabot's
PRs **don't** auto-merge, so it just produced competing/duplicate PRs
(e.g. the `actions/checkout` v6→v7 double-up in #36/#37). Renovate owns
everything now.

### 2. Add `helpers:pinGitHubActionDigests`
The repo's zizmor check enforces `unpinned-uses` (Actions must be
SHA-pinned). Renovate was proposing **bare tags** (`@v7`), which can
never pass zizmor → those PRs could never go green → never auto-merged.
Pinning to commit SHAs makes Action bumps pass zizmor **and** keeps them
updated (Renovate tracks the digest, with a version comment). Also a
supply-chain win.

## Effect
- One bot (Renovate), no duplicates.
- DNS server / base image / .NET / Debian package / Action /
devcontainer updates all create + auto-merge once CI is green.
- CI still gates every merge (Renovate won't merge a red PR — exactly
what stopped the broken .NET 10 bump from landing).

- 👷 Harden auto-release: SHA-pin checkout v7 & set make_latest (#39)

## Description

Two fixes to the auto-release workflow, plus it resolves the duplicate
`actions/checkout` bump PRs.

### 1. `make_latest=true` on publish
v15.3.0 published correctly but did **not** receive GitHub's "Latest"
badge (the API still returned v14.3.0). Cause: the workflow publishes by
PATCHing the draft with `draft=false`, but flipping a draft to published
via PATCH does not re-run GitHub's latest-release evaluation. Setting
`make_latest=true` explicitly fixes this for future releases.

### 2. `actions/checkout` v6 → v7 (SHA-pinned)
This is the only direct `actions/checkout` usage in the repo. Pinned to
the `v7.0.0` commit SHA to match the repo's existing pin convention and
satisfy zizmor's `unpinned-uses` rule — which is exactly why the
bare-tag bumps in #36 and #37 fail their zizmor check.

v7's breaking change (blocking fork-PR checkout in
`pull_request_target`/`workflow_run`) does not affect this step: it
specifies no `ref` and only runs on `push`-triggered runs, so it checks
out `main`, not untrusted PR code.
